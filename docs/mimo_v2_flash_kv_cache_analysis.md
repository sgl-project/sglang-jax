# MiMo-V2-Flash TPU v6e KV Cache 理论分析报告

## 1. 分析目标

在 TPU v6e (32 GB HBM/chip) 上推理 MiMo-V2-Flash (~309B 总参数)，计算 5 种 DP/TP/EP 切分策略下的：
- 每 chip 模型权重占用
- 可用 KV cache 内存
- 每请求 KV cache 大小
- 最大可支持并发 Batch Size (BS)

输入场景：4k+1k (5,120 tokens) 和 16k+1k (17,408 tokens)

## 2. 模型架构参数

| 参数 | 值 |
|------|-----|
| 总层数 | 48 |
| hidden_size | 4096 |
| Q heads | 64, head_dim=192 |
| **Full Attention 层** | **9 层** (0,5,11,17,23,29,35,41,47) |
| Full attn KV heads | 4, K_dim=192, V_dim=128 |
| **SWA 层** | **39 层** |
| SWA KV heads | 8, K_dim=192, V_dim=128 |
| sliding_window | 128 |
| MoE | 256 experts, top_k=8, intermediate=2048 |
| Dense FFN | 仅 layer 0, intermediate=16384 |
| 量化 | FP8 (e4m3), o_proj 保留 bf16 |
| vocab_size | 152,576 |

## 3. 运行时参数

| 参数 | 值 |
|------|-----|
| HBM/chip | 32 GB |
| KV cache dtype | bfloat16 (2 bytes) |
| mem_fraction_static | 0.88 |
| 可用预算 | 32 × 0.88 = **28.16 GB/chip** |
| EP 策略 | EP = total_chips (专家按个数切到所有芯片) |
| SWA 分析方式 | 理论最优：每请求仅存 128 tokens |

## 6. KV Cache 计算

### 6.1 head_dim 对齐

sgl-jax 代码将 head_dim 对齐到 128 的倍数 (`(dim + 127) // 128 * 128`)：

| 原始 dim | 对齐后 |
|----------|--------|
| K: 192 | **256** |
| V: 128 | **128** |

### 6.2 物理 KV Head 数

bfloat16 的 packing factor = 32 // 16 = 2，物理 head 需对齐到 2 的倍数：
`physical = ceil(logical / 2) × 2`

| TP | Full attn (kv=4) | SWA (kv=8) |
|----|---|---|
| 16 | logical=1 → **physical=2** (50% 浪费) | logical=1 → **physical=2** (50% 浪费) |
| 4 | logical=1 → **physical=2** (50% 浪费) | logical=2 → **physical=2** (0% 浪费) |
| 2 | logical=2 → **physical=2** (0% 浪费) | logical=4 → **physical=4** (0% 浪费) |

### 6.3 每 Token 每层 KV Cache (per chip)

公式: `physical_heads × (K_aligned + V_aligned) × 2 bytes`

| TP | Full Attn | SWA |
|----|-----------|-----|
| 16 | 2 × (256+128) × 2 = **1,536 B** | 2 × (256+128) × 2 = **1,536 B** |
| 4 | 2 × (256+128) × 2 = **1,536 B** | 2 × (256+128) × 2 = **1,536 B** |
| 2 | 2 × (256+128) × 2 = **1,536 B** | 4 × (256+128) × 2 = **3,072 B** |

### 6.4 每请求 KV Cache (per chip)

- Full attn (9 层): context_length × 1,536 × 9
- SWA (39 层): min(128, context_length) × swa_bytes × 39

| TP | 场景 | Full Attn (MiB) | SWA (MiB) | **总计 (MiB)** |
|----|------|-----------------|-----------|----------------|
| 16/4 | 4k+1k (5,120 tok) | 67.50 | 7.31 | **74.81** |
| 16/4 | 16k+1k (17,408 tok) | 229.50 | 7.31 | **236.81** |
| 2 | 4k+1k (5,120 tok) | 67.50 | 14.62 | **82.12** |
| 2 | 16k+1k (17,408 tok) | 229.50 | 14.62 | **244.12** |

> **SWA 窗口优势**: 39 层 SWA 仅占每请求 7~15 MiB，远小于 9 层 Full attn 的 68~230 MiB。
> sliding_window=128 是 MiMo-V2-Flash 内存效率的关键优势。

## 8. 深度分析

### 8.1 DP 倍增效应

DP 是提升总 BS 的最有效手段：

| 对比 | 同硬件 | BS/shard 差异 | DP 差异 | 总 BS 差异 |
|------|--------|-------------|---------|-----------|
| A→B | 16 chips | 135→79 (-41%) | 1→8 (8×) | 135→632 (**4.7×**) |
| A→C | 16 chips | 135→114 (-16%) | 1→4 (4×) | 135→456 (**3.4×**) |
| B→D | 16→32 chips | 79→189 (+139%) | 8→16 (2×) | 632→3,024 (**4.8×**) |

### 8.2 EP=32 vs EP=16 的影响

32 卡配置将专家权重减半 (17.63 → 8.81 GiB/chip)，释放 ~8.8 GiB 用于 KV cache：

| 对比 | 权重差 | KV 增加 | BS/shard 提升 |
|------|--------|---------|-------------|
| B(EP16) → D(EP32) | -8.81 GiB | +8.81 GiB | 79→189 (**2.4×**) |
| C(EP16) → E(EP32) | -8.81 GiB | +8.81 GiB | 114→235 (**2.1×**) |

### 8.3 TP=2 vs TP=4 权衡

| 维度 | TP=2 | TP=4 |
|------|------|------|
| 非专家权重/chip | 4.19 GiB | 2.14 GiB |
| SWA KV/token/layer | 3,072 B (physical=4) | 1,536 B (physical=2) |
| SWA KV/请求 (128 tok) | 14.62 MiB | 7.31 MiB |
| KV head padding 浪费 | 0% (full+SWA 均对齐) | 50% (full attn 1→2) |
| All-reduce 通信 | 2 chips, 低延迟 | 4 chips, 中延迟 |

TP=2 在非专家权重和 SWA KV 上开销更大，但因 SWA 窗口仅 128 tokens，差异有限 (~7 MiB/请求)。

### 8.4 KV Cache 构成分析

以 16k+1k 场景为例 (TP=4):

```
每请求 KV = 236.81 MiB

Full Attention (9层):  ████████████████████████████████████████████████░ 229.50 MiB (96.9%)
SWA (39层):            ██░ 7.31 MiB (3.1%)

Full Attention 占绝对主导，SWA 的 128-token 窗口几乎可以忽略不计。
```

## 9. 配置推荐

### 最大并发吞吐
- **Case D (32×DP16×TP2)**: 4k+1k 总 BS=3,024, 16k+1k 总 BS=1,008
- 适合吞吐优先、延迟不敏感的离线批处理场景

### 最佳性价比 (16 卡)
- **Case B (16×DP8×TP2)**: 总 BS 为 Case A 的 4.7×
- DP=8 的倍增效应远超 TP=2 的额外权重/KV 开销
- 适合中等规模部署

### 最优每分片容量
- **Case E (32×DP8×TP4)**: BS/shard=235 (4k+1k) 为所有 Case 最高
- 适合需要单分片高并发的场景 (如延迟敏感)

### 不推荐
- **Case A (16×TP16)**: DP=1 导致总 BS 最低，且 TP=16 通信开销极大

## 10. 注意事项

1. **理论上限**: 实际 BS 会低于理论值，因为需考虑 KV cache 碎片化、activation memory、radix cache 等
2. **SWA 实现差异**: 当前 sgl-jax 使用 ratio-based SWA 内存分配 (swa_full_tokens_ratio=0.8)，未做严格的 per-request 128-token 限制
3. **EP 通信开销**: EP=32 需要全芯片 all-to-all 通信，可能成为延迟瓶颈
4. **head padding**: bf16 packing=2 导致 full attn 在 TP ≥ 4 时有 50% KV head 浪费，可通过 FP8 KV cache (packing=4) 或调整 TP 来缓解
5. **o_proj bf16**: 所有 48 层 o_proj 保留 bf16，占非专家权重的主要部分 (TP=16 时占 ~30%)
6. **HBM 解释**: 本分析使用 32 GiB = 32 × 2^30 bytes 作为 HBM 基准
